## 2026-06-11

### Sound system alive + game art renders: HALESTORM driver runs natively

The big unlock chain, found by peeling one blocker after another:

- recomp: third Think C switch style (LEA d16(PC),An; ADDA.W (An,Dn.W*2),An;
  JMP (An)) with CMPI- or MOVEQ/CMP-form bounds checks. Unrecognized, it
  silently no-op'd the sound-system init dispatch, the music command
  dispatcher AND the blitter's 13-case run-mode dispatch.
- recomp: scan_jump_table extended to An-indexed JMP after LEA (d16,PC),An
  (the MDRV driver's 38-command entry dispatch).
- recomp: movep (driver banging VIA), chk, negx, ori/andi/eori to SR,
  MOVE to SR translated; raw low-mem Ticks (0x16A) reads become
  ticks_live() so the driver's CPU benchmark busy-waits see real time.
- MDRV 'MIDI Synth 3.45' resource: offline decoder (tools/decode_mdrv.py,
  LCG stream cipher seed 0xDCE5 + LZSS) -> recompiled as segment 25 at
  0x390000; the in-game unpack NewHandle is pinned to that base so the
  blob lands where the native code expects it.
- runtime: VBL tasks (VInstall/VRemove + 60Hz catch-up firing), Sound
  Manager channels + SndPlayDoubleBuffer ping-pong pump (the synth's
  audio clock), TM tasks get tmCount/tmWakeUp semantics (RmvTime reports
  unexpired time - the music tick stopwatch depends on it), OS traps set
  NZ from D0 on exit (callers BNE right after NewPtr etc).
- SysEnvirons: hasFPU/hasColorQD are BYTES; the word-sized overflow was
  smashing the caller's saved A6 (=> sound init read its params through
  a truncated frame pointer and failed with -11).
- SwapMMUMode returns the old mode MOVEQ-style (full D0): the blitter's
  32-bit row loop relies on the cleared high word; preserving it sprayed
  multi-MB fills across the heap (the 04-flood).
- NewGWorld ports get clipRgn = bounds (the game's blitter clips by
  clipRgn only); FSSpec resolver tolerates doubled path components;
  resource cache survives the game DisposeHandle-ing resources behind
  our back; CopyBits gets POP2_TRACE_BLT; memory manager gets a
  double-free detector and alloc-fail diagnostics; POP2_WATCH=hex traps
  guest writes to an address with a host backtrace.

full level art, sprites and status bar rendering (190 colors); the
HALESTORM synth initializes, loads instruments, plays the song clock

### SndChannel commands: bufferCmd duration + callBackCmd completion + SFX mix hook

Sound Manager channels were no-ops; the engine's f5_52c8(id) 'is sound
playing' poll reads slot flags that only a channel callback clears, so
any flow waiting on a one-shot SFX (death restart, jingles, story page
turns) could stall. Implement per-channel command queues: bufferCmd
tracks busy time from the stdSH SoundHeader and stashes PCM, callBackCmd
fires the guest callback (SndCommand written to chan+20) when the queue
drains, quiet/flushCmd reset. snd_chan_pump runs from snd_pump's tick
sites; sfx_mix_into mixes pending voices into outgoing music buffers.

Also: watch_hit (POP2_WATCH) now logs value changes only, with ticks.

Note: PoP2 gameplay SFX use a second path (seg20 channel objects ->
seg11 voice requests -> HALESTORM driver sample voices) which still
never completes — the driver render never consumes the queued voice;
that is the actual blocker for the death/restart flow and is the next
target.

### recomp: jump to entry when shared tails precede it — fixes 6 dead functions

local_walk pulls shared epilogues reached by backward branches (e.g. the
driver's 'bump slice counter; rts' tail at seg25:215a) into the function
body. Instructions are emitted in address order, so a function whose body
contains blocks below its entry point started executing the pulled-in
tail — f25_219e (the per-voice render of the HALESTORM driver) returned
through a foreign rts on entry, so driver sample voices were installed
but never mixed and never completed. Emit 'goto L<entry>' after the
prologue when min(body) < entry.

Revives f2_0f6e, f2_5020, f20_0204, f20_0822, f25_03f8, f25_219e.

### gworld: pad rowBytes +16 like 32-Bit QD; fixes story pages 4+ dissolve

The intro pages 4+ render through a table-driven dissolve (f15_3050).
The table is the shipped 'DSLV' 6400 resource in Prince2.opt — a cache
the game generates on first run (f15_3878 with AddResource/Write, id =
screen rowBytes*10 + mode) and reuses forever. Its baked source offsets
assume the 512x384 page GWorld has rowBytes 528 — real 32-Bit QuickDraw
padded GWorld rows 16 bytes past 4-byte alignment. Our NewGWorld gave
exactly 512, so every dissolve row sheared 16px further right, turning
Jaffar into deterministic diagonal stripes (decode and composition were
fine all along: the GWorld held perfect art).

All other consumers read rowBytes from the PixMap, so padding is safe;
verified attract pages 1-5 plus the full death/restart/Cmd+L/level-2
gameplay chain. Also: guest_trace prints stack args + f15_3050 table
forensics, jump_virtual misses loggable via POP2_TRACE_JV, CopyBits
trace shows row strides.

### traps: implement PPostEvent (0xA12F). PoP2 posts an event when the canvas is clicked; the trap was unimplemented, so `fatal()` → `abort()` killed the whole wasm instance (the tab froze, audio cut out, only a reload recovered — far worse than the native exit). ### investigation: the "HP drain" is intended level-14 finale logic, not a miscompile (retraction). The two entries above are wrong on this point and are corrected here. The reported "HP drains on loaded saves" was first dismissed, then blamed on an -O1/-O2 miscompile — which was used to justify reverting the web build to -O0. A code-level trace settles it. The runtime's POP2_WATCH writes a host backtrace whenever a watched guest address changes value; pointed at the HP word (guest 0x7AFC8 = a5-20536) with the level-14 finale save loaded, then run through addr2line on the native -O0 build, it shows: the live HP is written by a block-copy f4_4b02 (a5-20690 → the live kid block a5-20566) driven by f6_4382, and the *source* HP (a5-20660) is decremented by f2_54d6 (seg02.cpp:9306) — a general integrator HP = clamp(HP + delta[a5-20656], 0, cap[a5-20658]). Both f2_54d6 and f6_4382 carry an explicit env[(a5-20500)+8582] == 6 check — the level-14 finale flag — gating a call to f4_210a. So this is faithfully-recompiled finale machinery (the awakened shadow/double draining the prince), gated on the finale env: levels 5 and 10 never reach it, which is exactly why only the level-14 saves bleed HP. The GameFAQs/popuw walkthroughs agree the shadow transformation costs lives ("you will lose one life until your shadow is out"). Consequences: (1) the "-O1/-O2 miscompiles the HP" claim is retracted — the drain reproduces on -O0 native, so it was never codegen; -O2 (~12 MB, faster boot) is a viable follow-up once verified against native, though the build stays -O0 conservatively for now. (2) The hand-poked finale debug saves L14fat/L14f11 (12 flasks → double awakened) are removed from the bundled volume and the web data rebaked, so loading a bundled save no longer drops the player into the draining finale state; the Invincibility assist also pins HP for anyone exploring the real finale.

### assist: "+10 minutes of game time" (clock rewind). The user asked for a simpler alternative to infinite-time — a button that grants more time. PoP2's time limit is real-time (TickCount) based: there is no counted timer global to freeze. Four memory searches confirmed it — a rising-counter sweep found only the free-running 60 Hz frame counter (a5-21412/a5-21322); value-matching the save's @98 field (long assumed to be the timer) across Level 5/10/14 loads found no live global; and a full-save diff showed the save is byte-identical over 30 s of play, so @98 is a constant, not elapsed time. The clean lever is the clock itself: the game computes elapsed = TickCount − start, so rewinding TickCount grants time. ticks_now() now subtracts a static s_tick_offset, and time_add_minutes(n) (exported as pop2_add_time, wired to a "+10 minutes" button in the Assists panel) bumps the offset by n×3600 ticks, capped at raw−60 so TickCount can never underflow or stall. Verified in-browser: clicking the button rewinds the live low-mem Ticks (0x16A) — e.g. 3229→151 and 703→204 — without freezing the game (saving still works immediately after, which needs the main loop, not just the event pump). Caveat: the game persists no timer field and the limit only gates the ending, so the end-to-end effect on the ending could not be verified headlessly; this is the correct lever for a TickCount-based limit and is harmless if no limit fires. The full freeze-then-infinite approach would instead need the time-limit *check* located in the ending code — deferred.

### assist: game-speed slider (0.25×–3×). The same insight that powered the +time button — the game derives all its pacing from TickCount (ticks_now) — makes a clean speed control. ticks_now() now advances a speed-scaled accumulator instead of returning raw ticks: each real tick adds s_speed game-ticks (s_scaled += (raw−last)×s_speed), which stays monotonic across speed changes so there is no jump. time_set_speed(mult) (exported pop2_set_speed, wired to an Assists-panel slider) sets the multiplier; it banks the time elapsed at the old speed before switching. The real-time present throttle (SDL_GetTicks, the 16 ms gate in video_pump) is untouched, so the display stays 60fps — high speeds fast-forward the logic shown at 60fps, low speeds are smooth slow-motion. Verified in-browser by timing the game's own frame counter (a5-21412) over 5 s: 1.0×→60.6/s, 2.0×→120.8/s (1.99×), 0.5×→30.2/s (0.50×), back to 1.0×→60.2/s, and the slider UI drives it identically (the rate and the value label track the handle). Composes with the +time offset (both adjust the same clock: t = scaled_ticks − offset). (An earlier draft of this entry wrongly said the tempo follows; it does not, which is arguably nicer — no chipmunk pitch-shift.) Making it follow would mean driving snd_pump from the scaled clock — not done, since normal-tempo audio under a speed change is the better default.

### fix: the real load freeze — the per-frame render gate stays off after a save-load. ### fix: alt-tab froze the game — a no-char menu key opened the About dialog into an unimplemented trap. Reported as "switching to another window stops the sound and freezes the game for good — New Game doesn't help, only a reload". The user's console log was decisive: `[menu] MenuKey('') -> 400:1` → `[dlg] GetNewDialog(600)` → `FATAL: unimplemented Toolbox trap _FrameRoundRect (0xA8B0)` → `Aborted(native code called abort())`. ## 2026-06-23 — Platform assist: reachability solver, per-level verification

Autonomous pass to actually *traverse* each level rather than just fill pits by
geometry. New `tools/level_solve.py` lays a level's linked rooms into one global
tile grid and simulates the prince: walk, climb up one, drop (<=3 free, deeper is
a fall), and -- for the intended-path model only -- jump a few tiles. It computes
reachability from the real start cell (collected via `POP2_DUMP_KID`) two ways:
`R_full` (jumps + survivable falls = where the designer lets you go) and `R_safe`

blanket pit-fills, **vets each one** (a fill is dropped if it shrinks `R_full` --

`R_safe` covers `R_full`. So it can't create a softlock by construction.

level: the solver's vetted+completed set where the movement model actually
traverses the level (its `R_full` is a big fraction of the walkable tiles), else
the geometry heuristic where the model gets stuck. Honest limitation: the model's
**vertical** reach (climbing/falling across the 3-row room seams) does not yet
match PoP2's real physics, so levels whose start chamber only exits up/down --
**L2, L6, L11, L14** -- the model can't leave (reachable 5-45 of ~150-190 cells),
and they fall back to geometry (not movement-verified). The other ten traverse
well (e.g. L8 237/237 reachable no-jump, L1 85/85). Per-level `# L N:` comments

## 2026-06-24 — Web perf: -O2 + a 60 Hz frame-limiter (main-thread CPU 100% → ~10%)

## 2026-06-25 (night) — Smooth + low-CPU web pacing: idle the VBL wait ("tick" mode), branch `pacing`

The 100% CPU was the recompiled engine's VBL busy-wait — a tight `while(TickCount()<n){}`
loop, each iteration pumping video — NOT the render work, so the lever is to idle that wait.
Earlier idle attempts regressed smoothness because they kept the 16 ms SDL_GetTicks present
gate, whose interval drifts against the 16.67 ms vsync → a periodic dropped frame (the user's
"рывки"). New approach presents once per frame with NO competing gate:

The trap dispatcher (do_trap) detects the VBL wait as a run of **consecutive poll traps**
(TickCount / WaitNextEvent / GetKeys / Button / GetMouse / SystemTask). Game logic interleaves
drawing and arithmetic traps, so its consecutive run stays short (measured < 16); only the idle
wait accumulates. After 32 consecutive poll traps the finished frame is presented and the thread
idles to the next animation frame (rAF). A 100 ms safety net forces a present if detection ever
misses, so it can't hard-freeze. Lower threshold = more robust under load (a heavy frame leaves a
short wait, and a low bar still clears it; tick=128 misses it and judders). A pump-count "spin"
mode was tried first but only works in a fragile 512–1024 window — superseded by tick.

Headless (real-compositor, Level 10; present-interval jitter is the judder signature):
- busy (current default):  CPU ~100%   ~60 fps   sd ~1 ms
- tick=32 (new):           CPU ~8–11%  ~60 fps   sd ~1 ms, 0 long gaps   (when the host is idle)
The CPU drop (~10x) is reliable across every run. The fps/jitter of tick is rAF-paced, so the
HEADLESS number is sensitive to host CPU contention (the build agent itself perturbed it: some
runs showed 30–40 fps while a competing process ran, others a flat 60). Real hardware locks rAF
to vsync, so that variance is a test-env artifact — but it does mean headless can't *prove*
smoothness; the user confirms by eye. Engine stays correct under tick: tickRate 60/s, +10 lands
on the HUD timer, save/load fine (toggle_test + addtime/time/plat all green).

Selectable at runtime so it can be A/B'd by eye: **default stays busy** (deployed build
unchanged); a new "Low-CPU mode" Assists toggle (`pop2_set_pace`) and `#pace=tick` switch to
tick; `#pace=spin&spin=N` / `#pace=oldraf` kept for debugging. Headless jitter instrumentation
added behind #fps ([jitter] lines). On branch `pacing` only — master/web keep the smooth
busy-loop until the user confirms tick is smooth on real hardware. Tooling:
~/pop2-webtest/jitter_run.mjs (CPU + jitter + fps + alive; STRESS=1 drives movement), toggle_test.mjs.

### web: fix the crackling cutscene audio (the real cause — the 100 ms pacing fallback).
Player confirmed the crackle is *only* in cutscenes; in-level continuous audio is clean — so the
audio pipeline is fine, something specific to cutscenes starves it. Read SDL2's Emscripten audio
backend: it plays through a `createScriptProcessor` node whose `onaudioprocess` runs on the MAIN
thread, sized by `spec.samples` (so the 512→2048 bump did take effect, ~42 ms). The smoking gun
is in `video_pump`: in tick mode the per-VBL idle is driven from the trap dispatcher's poll-trap
detection, and `video_pump` itself only force-presents as a **100 ms** safety net. NIS cutscenes
don't busy-wait the VBL the same way, so they fall through to that 100 ms fallback — the main
thread blocks ~100 ms between yields, far longer than the ~42 ms audio buffer, so it drains and
the voice/music crackle continuously (and the cutscene renders ~10 fps). A bigger buffer can't
win against a 100 ms block. Fix: drop the fallback to **24 ms** — well inside the buffer window,
